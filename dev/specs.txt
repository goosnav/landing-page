======================================================================
0. DOCUMENT PURPOSE
===================

This document defines the full software specification for a reusable,
enterprise-grade static website template for service businesses.

This is not a one-off website. This is a hardened core template intended
to be duplicated and rapidly customized for many different businesses.

The system must satisfy all of the following simultaneously:

* extremely clean architecture
* minimal stack complexity
* static hosting compatibility
* fast deployment
* high legibility for developers
* strong modifiability through centralized config/data files
* complete core business-site conventions
* no trendy UI clutter
* no framework dependency
* no unnecessary abstractions
* desktop-first but mobile-safe
* dark-theme by default
* easy adaptation to different brands, services, and quote logic

A competent developer must be able to read this document and build the
system completely without needing missing assumptions filled in later.

======================================================================

1. PRIMARY OBJECTIVES
   ======================================================================

1.1 PRODUCT OBJECTIVE

Build a static, multi-page, config-driven service business website
template with the following public pages:

* Home
* Services
* Quote
* About
* Contact
* Privacy
* Terms

The system must also include the following hidden placeholder pages:

* Articles / White Papers
* Misc

1.2 BUSINESS OBJECTIVE

Enable rapid cloning and customization of the template for different
businesses with minimal structural rewrites.

1.3 ENGINEERING OBJECTIVE

Enforce strict separation of concerns:

* HTML = page structure and semantic layout
* CSS = design system, layout, theme, components, utilities
* JS = behavior, rendering, validation, quote flow, summary generation
* Config/Data = brand identity, nav visibility, service tiers,
  testimonials, forms, quote rules, scheduler settings, SEO defaults

1.4 UX OBJECTIVE

The site must feel:

* corporate
* technical
* mature
* direct
* minimal
* trustworthy
* fast
* legible

The site must not feel:

* cute
* trendy
* gimmicky
* bubbly
* startup-template-like
* over-animated
* bloated
* CMS-heavy

======================================================================
2. PRODUCT PRINCIPLES
=====================

2.1 NO BLOAT

Every file, module, config object, section, and page must justify its
existence. Nothing decorative should exist solely because templates
usually include it.

2.2 STATIC FIRST

The site itself must remain static. Dynamic behavior is client-side JS
only. External services may be used only where static hosting cannot
perform the required secure function.

2.3 CONFIG FIRST

Common modifications must happen primarily in config/data files, not by
editing structural HTML repeatedly across the site.

2.4 LEGIBILITY OVER CLEVERNESS

Code must be obvious. Avoid abstraction for abstraction’s sake.

2.5 SEMANTIC HTML

Prefer standard HTML5 structure:

* header
* nav
* main
* section
* article
* aside
* footer
* form
* fieldset
* legend
* label
* button
* dialog only if truly needed

2.6 DESKTOP PRIORITY, MOBILE SAFETY

Desktop is primary. Mobile must remain coherent, touch-safe, readable,
and fully functional.

2.7 HARDENED TEMPLATE

This must be stable enough to serve as the reusable bedrock for serious
commercial websites, not a throwaway demo scaffold.

======================================================================
5. HIGH-LEVEL ARCHITECTURE
==========================

5.1 ARCHITECTURAL MODEL

The system is a static multi-page site with a config-driven rendering
layer and client-side interaction layer.

LAYER 1: STRUCTURE

* semantic page shells in HTML

LAYER 2: PRESENTATION

* CSS tokens, layout, theme rules, component rules, print rules

LAYER 3: BEHAVIOR

* JS for nav state, content rendering, quote engine, validation,
  scheduler handling, downloadable quote generation

LAYER 4: CONFIG/DATA

* JS config modules defining brand identity, page toggles, service data,
  testimonials, form endpoints, quote questions/rules, scheduler
  behavior, SEO defaults, article placeholders, misc placeholders

LAYER 5: EXTERNAL STATIC-COMPATIBLE SERVICES

* Formspree
* Google Calendar appointment schedule
* GitHub Pages

5.2 ARCHITECTURAL BENEFITS

* no runtime server to maintain
* trivial hosting
* low operational cost
* easy Git version control
* easy cloning per business
* very fast page loads
* low dependency risk
* straightforward debugging
* clear ownership boundaries

21. QUOTE WIZARD SPECIFICATION
==============================

21.1 CORE UX MODEL

The quote tool must behave like a slideshow:

* one question per screen
* next/back navigation
* visible progress
* smooth but restrained transitions
* no multi-question clutter per screen by default

21.2 FLOW STAGES

STAGE A: INTRO

* brief explanation
* estimated duration
* start button

STAGE B: QUESTION FLOW

* one question per screen
* answer capture
* branching based on prior answers
* next/back
* validation before advance

STAGE C: LEAD GATE

* must occur before showing results

STAGE D: RESULT

* estimated price range
* recommended tier or engagement type
* detailed service/scope breakdown
* assumptions
* next steps
* scheduler section
* downloadable summary

21.3 LEAD GATE REQUIREMENTS

At least one of the following must be provided:

* full name
* company name

AND

At least one of the following must be provided:

* email
* phone

Opt-in checkbox:

* optional
* not pre-checked
* phrased politely and commercially
* stored as part of submission if checked

Recommended label:
“Keep me informed about useful updates, relevant service offerings, and
occasional insights.”

21.4 QUOTE QUESTION TYPES

The engine must support at minimum:

* single select buttons
* radio group
* multi-select checkbox group
* dropdown select
* number input
* short text input
* yes/no
* date estimate optional
* textarea optional

21.5 QUESTION DATA MODEL

21.10 BACK BUTTON BEHAVIOR

* basic client-side validation
* rate-limiting delegated to provider

24.6 QUOTE LEAD SUBMISSION TIMING

Lead submission should occur:

* after questionnaire completion
* at lead gate submit
* before result is rendered

If submission fails:

* show user-friendly error
* allow retry
* do not silently discard lead gate data

31. SECURITY / RISK BASELINE
============================

31.1 CORE SECURITY POSTURE

This is a static site. Its security model is largely:

* minimal moving parts
* no backend code to harden in repo
* secure external providers for forms/scheduling

31.2 SECURITY RULES

* do not place secrets in the repo
* do not place API keys in client JS
* assume all front-end logic is visible
* do not claim quote logic is hidden
* do not over-collect sensitive data
* minimize localStorage use
* sanitize rendered text if any HTML injection path is allowed

31.3 PRIVACY RULES

* collect only information actually needed
* keep consent checkbox optional and explicit
* provide privacy page
* avoid unnecessary persistent client-side storage
