DELIVERY STYLE: LOW-BLOAT, HIGH-LEGIBILITY, TEST-DRIVEN EXECUTION

======================================================================
0. EXECUTION MODEL
======================================================================

This plan assumes disciplined engineering execution, not ad hoc page
building.

The sprints are arranged to de-risk the project in the correct order:

1. architecture first
2. design system second
3. shared shell and navigation third
4. core public pages next
5. data/config-driven rendering after structure is stable
6. quote engine after public shell is mature
7. external integrations after internal behavior is stable
8. hardening, SEO, accessibility, and documentation last

This is the correct order because:
- it minimizes rework
- it prevents “template rot”
- it forces core abstractions to prove themselves early
- it preserves simplicity
- it keeps styling and logic from becoming entangled

Sprint duration assumption:
- 1 sprint = 1 focused work block of roughly 4-7 engineering days
- a solo engineer could compress or expand this as needed
- the order should not materially change

Each sprint contains:
- objective
- scope
- implementation tasks
- test requirements
- exit criteria
- failure modes to watch

======================================================================
1. ENGINEERING RULES FOR ALL SPRINTS
======================================================================

These rules apply across the full project.

1. No framework creep.
2. No speculative features.
3. No duplicated business content unless intentionally page-specific.
4. No hardcoded services/testimonials/contact data scattered across pages.
5. No unowned CSS hacks.
6. No hidden global JS state.
7. No adding features without defining test coverage.
8. Every sprint must end in a working, reviewable state.
9. Every sprint must preserve GitHub Pages compatibility.
10. Every sprint must improve the template, not just one instance.

Required quality bars for every sprint:
- works on desktop
- remains usable on mobile
- has no broken nav paths
- no console errors
- no layout explosions at standard viewport widths
- code is commented where future modifiers would otherwise get confused

19. FINAL ENGINEERING JUDGMENT
======================================================================

The mistake junior teams make is building pages first, logic second, and
reuse never. That always produces a fragile website.

The correct senior-engineer approach is:

- build the bedrock
- lock the contracts
- prove the shell
- prove the config model
- prove the service model
- isolate the quote engine
- integrate external services only after internal logic is stable
- harden accessibility, metadata, performance, and documentation last

That sequence produces a reusable asset, not a one-time artifact.

This sprint plan is optimized for that outcome.
